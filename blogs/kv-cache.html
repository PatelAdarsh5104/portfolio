🚀 KV Cache in LLMs — What it is, Why it matters, and How you already use it

Published on April 26, 2024

🔹 What is KV Cache & Why do we need it?

LLMs generate text one token at a time. For every new token, the model looks back at all previous tokens using the attention mechanism.

Without optimization:

  • The model recomputes attention for the entire history
  • This happens again and again
  • Even though the past context never changes

Root Cause: We are redoing expensive computation for old tokens when we don’t need to.

That’s where KV Cache comes in. KV Cache means storing the “Keys” and “Values” of past tokens so the model doesn’t recompute them. In short: "Don’t redo work you’ve already done."

🔹 How does KV Cache help? (Speed + Cost)

Because past tokens are cached, we get:

  • ✅ Faster inference: Each new token only computes attention for itself. Latency doesn’t grow linearly with conversation length.
  • ✅ Lower cost: Less compute per request and fewer GPU cycles. Especially important for long prompts and chat apps.

🔹 Tiny example

Imagine a chat like this:

  1. “Here is a long API document…” (2000 tokens)
  2. “How does authentication work?”
  3. “What about rate limits?”

Without cache: The 2000-token document is reprocessed every time.
With cache: The document is processed once; follow-up questions reuse it. Much faster and cheaper.

🔹 How do we use KV Cache today? (OpenAI & Bedrock)

The best part? You probably don't even have to manage it manually.

🚀 OpenAI

  • Prompt caching / internal KV reuse is enabled automatically.
  • Repeated prompt prefixes result in faster and cheaper requests.
  • It's handled entirely on the backend.

🚀 Amazon Bedrock (Claude 3.5 Sonnet)

  • Supports Prompt Caching.
  • You can mark stable parts of the prompt (docs, system rules) as cacheable.
  • Bedrock handles the rest.

🔹 Final Takeaway

  • Problem: Recomputing attention for past tokens is wasteful.
  • Root cause: Past context doesn’t change.
  • Solution: Cache it (KV Cache / Prompt Cache).
  • Result: Faster inference + lower cost.
  • Reality: OpenAI & Bedrock already do this for you.

Understanding this helps you design better prompts, build cheaper LLM apps, and reason about performance at scale.
